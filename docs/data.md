# 资料搜查指南（面向中国大陆消费维权 RAG 知识库）

## 1. 目标

你要准备的不是“越多越好”的资料，而是“能稳定支持检索和引用”的资料。

MVP 阶段只收四类：

1. 法条
2. 维权流程/规则
3. 案例摘要
4. 文书模板

原则：

- 官方优先
- 现行有效优先
- 少量高质量优先
- 可结构化优先
- 能明确引用来源优先

---

## 2. 需要什么资料

### 2.1 法条资料

这是知识库的核心。

优先收：

- 《中华人民共和国消费者权益保护法》
- 《中华人民共和国消费者权益保护法实施条例》
- 《中华人民共和国电子商务法》
- 《中华人民共和国民法典》中与合同、格式条款、违约责任、退款退货相关的条文
- 《中华人民共和国民事诉讼法》中与起诉、受理、证据、管辖相关的基础条文

建议扩展：

- 与市场监管投诉举报相关的部门规章
- 与网络交易、平台责任相关的规范
- 与预付式消费、教育培训、网络购物相关的专项规则

你最终要保留的不是整站资料，而是：

- 法律名称
- 条号
- 条文正文
- 发布机关
- 公布/施行时间
- 来源链接
- 当前是否有效

---

### 2.2 维权流程/规则资料

用于回答“该怎么做”。

优先收：

- 12315 投诉流程说明
- 市场监管投诉举报处理规则
- 主要电商平台公开售后/退货/争议处理规则
- 网上立案流程说明
- 起诉材料要求

保留重点：

- 适用场景
- 办理步骤
- 所需材料
- 办理时限
- 受理条件
- 来源链接

---

### 2.3 案例资料

MVP 不建议收海量原始裁判文书全文。

建议只收“精选案例摘要”，覆盖高频消费场景：

- 退款退货
- 商品质量问题
- 虚假宣传/货不对板
- 预付式消费退费
- 平台责任

每个案例至少整理成：

- 案件名称或编号
- 法院
- 日期
- 事实摘要
- 争议焦点
- 裁判要点
- 结果摘要
- 涉及法条
- 来源链接

原则：

- 优先选有明确裁判规则意义的案例
- 优先选与你想覆盖的场景强相关的案例
- 不需要多，先每类 10 到 20 个即可

---

### 2.4 文书模板资料

用于后续模板化生成。

MVP 建议只准备：

- 投诉信
- 平台申诉文案
- 催告函
- 民事起诉状草稿

每个模板建议拆成：

- 使用场景
- 必填信息项
- 固定结构
- 可变段落
- 常用诉求表述
- 常用法条引用位置

不要只留一篇最终成文，最好同时整理出“模板字段表”。

---

## 3. 去哪找

### 3.1 法条

优先级从高到低：

1. 国家法律法规数据库
2. 中国政府网
3. 全国人大网
4. 最高人民法院官网（司法解释、规则）
5. 国家市场监督管理总局官网

用途分工：

- 国家法律法规数据库：法条主库
- 中国政府网：行政法规、条例
- 最高法官网：司法解释、案例规则
- 市监总局官网：投诉举报处理办法、消费监管规则

---

### 3.2 维权流程/规则

优先找：

- 全国 12315 平台及其官方说明
- 国家市场监督管理总局官网
- 各大平台官方帮助中心/规则中心
- 人民法院在线服务、法院官网立案指引

注意：

- 平台规则只能找官方公开版本
- 不要用二手转载作为正式入库版本

---

### 3.3 案例

优先找：

- 人民法院案例库
- 中国裁判文书网
- 最高人民法院官网发布的典型案例、指导性案例、参考案例

建议做法：

- 原文可以来自公开案例平台
- 入库时自己整理成摘要版
- 不直接把全文作为主检索单元

---

### 3.4 模板

优先来源：

- 你自己根据业务需要整理
- 法院公开文书格式要求
- 官方公开诉讼文书样式
- 你认可的内部模板

原则：

- 模板以可填写、可参数化为目标
- 不追求华丽措辞，追求结构稳定

---

## 4. 找什么格式的资料

### 最推荐格式

按优先级：

1. Markdown（.md）
2. JSON（结构化）
3. TXT
4. DOCX
5. PDF

原因：

- md / txt / json 最容易清洗和切分
- docx 次之
- PDF 最麻烦，尤其扫描版和复杂排版

### 每类资料推荐格式

- 法条：md / txt / json
- 规则：md / txt / docx
- 案例摘要：json / md
- 模板：json + md

### 案例摘要推荐 JSON 结构

```json
{
  "title": "某某消费者与某平台网络购物合同纠纷案",
  "court": "某某人民法院",
  "date": "2024-05-01",
  "facts": "...",
  "issue": "...",
  "holding": "...",
  "result": "...",
  "laws": ["消费者权益保护法第24条"],
  "source_url": "..."
}
````

### 模板推荐 JSON 结构

```json
{
  "template_type": "complaint_letter",
  "scene": "quality_dispute",
  "required_fields": [
    "consumer_name",
    "merchant_name",
    "transaction_time",
    "amount",
    "issue",
    "claim"
  ],
  "template_text": "..."
}
```

## 5. 你搜资料时要遵守的筛选规则

### 必须满足

* 来源明确
* 版本明确
* 时间明确
* 文本可复制
* 内容完整
* 能定位到条号/章节/案例编号

### 尽量避免

* 扫描图片 PDF
* 没有来源链接的转载
* 旧版本但没标注失效时间的法规
* 排版混乱、复制后乱码的网页
* 没有结构的长篇原始裁判文书全文

---

## 6. 建议的资料整理方式

### 建议目录

```text
materials/
  laws/
  rules/
  cases/
  templates/
  metadata/
```

### 建议命名方式

文件名尽量稳定、可读：

* `law_消费者权益保护法_现行.md`
* `law_电子商务法_现行.md`
* `rule_12315投诉流程.md`
* `case_网购退款_某案_2024.json`
* `template_投诉信_商品质量.md`

```

### 7. 你交给系统前的最低整理标准

每份资料至少补齐这些字段：

- `title`
- `doc_type`
- `source_name`
- `source_url`
- `effective_date`
- `version_note`
- `raw_text`

推荐 `doc_type`：

- `law`
- `rule`
- `case`
- `template`

---

## 8. MVP 资料量建议

不要一开始准备太多。

建议起步量：

- 法条：20 到 80 条核心条文
- 规则：10 到 30 段流程规则
- 案例摘要：30 到 60 个
- 模板：4 到 8 个

这个量已经足够做第一版 RAG 验证。

---

## 9. 最后检查清单

在交给 Codex / 系统前，逐项确认：

- [ ] 是否都是你认可的资料
- [ ] 是否优先采用官方来源
- [ ] 是否尽量不是 PDF 扫描件
- [ ] 是否保留来源链接
- [ ] 是否保留时间/版本信息
- [ ] 法条是否能按条定位
- [ ] 案例是否已经整理成摘要
- [ ] 模板是否已拆成可填写结构

---

## 10. 一句话原则

你给系统的，不应该是一堆“文件”，而应该是一批“可切分、可检索、可引用”的材料。
```
